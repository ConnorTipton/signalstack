SignalStack V1 - Budget Blueprint
Version: 1.1
Purpose: This file replaces the original SignalStack V1 blueprint with a lower-cost operating design while preserving the same core goal: a personal options-alert research engine focused on signal quality, tradability, and replayability.

==================================================
1. PROJECT GOAL
==================================================
Build a personal options-alert research engine that monitors a small set of liquid U.S. large-cap stocks and ETFs in real time and sends high-conviction bullish or bearish options alerts only when all three conditions align:
1) a meaningful news catalyst,
2) price confirmation,
3) unusual or strongly suggestive directional options activity.

This is a personal research and paper-trading tool first. It is not a public product, not a social trading app, and not a fully autonomous trading bot in V1.

Primary objective:
- Maximize signal quality, tradability, and reviewability.

Secondary objective:
- Build a system that can later be extended with stronger paid data, more symbols, additional structures, and live execution.

Budget objective:
- Keep normal monthly operating cost under $100 by default.

==================================================
2. V1 SCOPE
==================================================
In scope:
- 20 to 30 liquid large-cap stocks and ETFs.
- U.S. regular market hours only.
- Bullish and bearish alerts.
- Long calls and long puts only.
- Next-week expiration by default.
- Same-week expiration only after paper validation.
- Low-cost real-time or near-real-time market/options data depending on provider mode.
- Real-time or near-real-time news from official feeds plus a low-cost aggregator.
- Telegram alerts.
- Optional paper order routing.
- Full logging and replay.
- Provider confidence tracking.

Out of scope:
- Live trading automation.
- X/Twitter sentiment as a primary detector.
- Discord/community features.
- Multi-leg spreads.
- Mobile app.
- Public SaaS/productization.
- Black-box LLM-only trade decisions.
- Enterprise-grade news/data spend in V1.

==================================================
3. CORE DESIGN PRINCIPLES
==================================================
A. Liquidity first
Choose highly liquid underlyings and contracts with acceptable spreads, visible activity, and realistic fills.

B. Rules first, LLM second
The signal trigger must be rule-based. The LLM should summarize, classify, and explain, not freely invent trades.

C. Official news first
Prefer direct issuer, SEC, exchange, and major wire releases over social chatter and scraped noise.

D. Every alert must be decision-ready
Each alert should include the why, the contract idea, the entry condition, invalidation, target, and time stop.

E. Paper first
Validate the workflow with alerts and paper execution before any live trading.

F. Replay matters
The system must store enough raw and normalized data to replay historical sessions exactly as the engine saw them.

G. Graceful degradation
Cheap sources are uneven. The system must keep working when a provider is weaker, but it must tighten thresholds and downgrade alert confidence when source quality falls.

==================================================
5. WHY THIS STACK
==================================================
Tradier:
Strong low-cost fit if available because it provides real-time U.S. stock and options market data to brokerage account holders, option chains, streaming support, and a sandbox environment. It is the best default replacement for an expensive all-in-one options data vendor.

Alpaca:
Best used as the paper-execution layer and as a secondary market/news adapter. Alpaca is also useful as a fallback market-data source during development and as a future paid upgrade path if better options coverage becomes necessary.

Official feeds + Marketaux:
Official feeds are the cleanest low-cost way to detect real catalysts. Marketaux broadens coverage across many sources and reduces the chance of missing relevant headlines between official releases.

Local LLMs:
A local model can handle routine headline labeling and short explanations cheaply. A tiny cloud budget should only be used for edge cases, retries, or quality checks.

Python:
Best fit for data ingestion, event processing, APIs, testing, and integration with multiple lower-cost providers.

7. OPERATING MODES
==================================================
The system should support three explicit modes.

Mode A: Build mode
Use when minimizing spend and validating plumbing.
- Alpaca free market/news where helpful,
- official news feeds,
- local LLM only,
- dry alerts only,
- options abnormality can run in proxy mode,
- no paper orders.

Mode B: Core budget mode
Default V1 target.
- Tradier real-time market/options data,
- official news feeds,
- Marketaux Basic or Standard,
- local LLM plus tiny cloud fallback,
- Telegram alerts,
- optional Alpaca paper routing.

Mode C: Upgrade mode
Use only after clear evidence the cheap stack is missing too many tradeable setups.
- keep the same architecture,
- swap in stronger paid options data and/or stronger paid news,
- preserve all normalized schemas and replay assumptions.

==================================================
8. SYSTEM ARCHITECTURE
==================================================
There should be one repository with three main runtime roles.

1. Ingestion workers
- connect to Tradier, Alpaca, Marketaux, and official feed sources,
- normalize provider events,
- store raw payloads,
- store normalized rows,
- handle reconnect/retry/dedup,
- label each event with source_tier and provider_confidence.

2. Signal engine worker
- read new events,
- run detectors,
- score candidates,
- select contracts,
- create alerts,
- optionally send paper orders to Alpaca,
- downgrade or suppress alerts when data confidence is weak.

3. Review/API service
- expose recent alerts,
- expose health checks,
- expose daily performance and symbol-level review,
- support replay/reporting workflows,
- expose provider health and ingestion lag.

==================================================
9. PROVIDER HIERARCHY
==================================================
Build adapter interfaces so providers can be swapped without rewriting detector logic.

Market/options adapter priority:
1. Tradier live
2. Alpaca live/free
3. recorded replay data

News adapter priority:
1. Official source feeds
2. Marketaux
3. Alpaca news
4. StockNewsAPI optional

Source tiers for news:
- Tier 1: SEC filings, issuer IR releases, exchange notices, top-tier direct wire releases.
- Tier 2: reputable aggregated articles with clear ticker mapping.
- Tier 3: backup/general market articles with weaker specificity.

Best practice:
- detector logic should consume normalized events, never provider-specific payloads directly,
- all provider routing decisions should live in adapters or router modules,
- each alert should record which sources contributed to the decision.

==================================================
12. NEWS INGESTION DESIGN
==================================================
The news layer is now multi-source by design.

Pipeline:
1. Fetch official feeds first.
2. Fetch aggregator sources second.
3. Normalize headline/body/URL/tickers/published time.
4. Deduplicate across sources.
5. Run rules-based prefilter.
6. Run LLM labeling only on candidates that survive prefilter.
7. Promote the event to detector input.

News dedupe keys:
- exact URL match,
- normalized title hash,
- same ticker cluster plus close publish time,
- same wire item updated by a second source.

News confidence rules:
- Tier 1 official source can pass with one strong item.
- Tier 2 aggregated item should usually require either a second confirming source or strong price confirmation.
- Tier 3 items should almost never trigger by themselves.

LLM should not read everything.
Use keyword/rules prefilters first:
- earnings,
- guidance,
- SEC filing,
- FDA/regulatory,
- analyst action,
- M&A/partnership,
- recall/litigation,
- product launch,
- executive change,
- major macro exposure tied to a specific ticker.

==================================================
13. DETECTOR LAYER
==================================================
An alert still requires alignment across three detectors, but the cheaper stack changes how confidence is handled.

Detector A: News catalyst
Purpose:
- identify meaningful bullish or bearish catalysts.

Required outputs:
- ticker
- event_type
- polarity
- importance
- confidence
- source_tier
- one_sentence_summary

Example event types:
- earnings beat/miss
- guidance raise/cut
- analyst upgrade/downgrade
- partnership/M&A
- legal/regulatory
- product launch
- company-specific macro-sensitive event

Detector B: Price confirmation
Purpose:
- refuse to alert on headlines that do not produce real price confirmation.

Examples:
- break of first 5-minute high after bullish catalyst,
- reclaim and hold above VWAP,
- loss of support after bearish catalyst,
- failed bounce and downside continuation.

Detector C: Options activity
Purpose:
- confirm that options activity is abnormally directional.

Full mode examples:
- call/put trade burst vs rolling baseline,
- elevated notional concentration,
- unusual activity in one expiry/strike,
- quote activity plus trade activity increasing together,
- liquidity remains acceptable.

Proxy mode examples:
- concentration of chain attention in one expiry/strike,
- sudden spread improvement plus activity pickup,
- same-direction contract clustering,
- near-the-money next-week interest rising quickly,
- option activity signal is suggestive but not fully confirmed.

Cheap-stack rule:
- if options detector is only in proxy mode, the system may issue a lower-confidence alert or watch alert,
- high-priority alerts should require stronger options evidence.

==================================================
14. SIGNAL SCORING
==================================================
Use a transparent weighted score.

Suggested initial score:
- news quality: 35
- price confirmation: 30
- options abnormality: 20
- liquidity quality: 10
- data confidence: 5

Suggested thresholds:
- 82 and above = high-priority alert
- 72 to 81 = valid alert
- 65 to 71 = watch alert or log-only depending on mode
- below 65 = log only, no alert

Additional caps:
- proxy-mode options data cannot produce high-priority alerts unless news is Tier 1 and price confirmation is strong,
- Tier 3 news cannot produce an alert without exceptional price and options confirmation,
- weak provider confidence should cap the maximum alert grade.

Every rejected candidate must store a rejection reason such as:
- weak catalyst,
- no price confirmation,
- options activity not unusual enough,
- options data only suggestive,
- contract too illiquid,
- signal too late in session,
- provider confidence too low.

==================================================
15. CONTRACT SELECTION RULES
==================================================
V1 contract selection should stay simple and disciplined.

Default rules:
- next-week expiration by default,
- near-the-money or one strike in-the-money,
- reject wide spreads,
- prefer visible open interest,
- prefer same-day trading activity,
- reject dead or hard-to-fill chains.

Additional budget-stack rules:
- if quote quality is uncertain, prefer the most liquid obvious strikes,
- if spread data is unreliable, downgrade the signal rather than forcing a contract pick,
- contract selection must record why a contract was chosen and why alternatives were rejected.

Why next-week first:
- gives the alert engine more room for imperfect timing,
- reduces fragility relative to very short-dated contracts.

Why single-leg first:
- easier to evaluate signal quality without the extra variables of spreads.

==================================================
16. ALERT FORMAT
==================================================
Every alert should be decision-ready.

Template:
Ticker | Direction | Score | Confidence Grade
Reason: short explanation of catalyst + price + options confirmation
Contract: expiration + strike + call/put
Entry: the price condition that must remain valid
Invalidation: what breaks the setup
Target 1: first trim level
Target 2: final target or exit condition
Time stop: when to close if no follow-through
Liquidity: quick note on spread/quality
Data note: provider mode and any confidence caveat

Example:
AAPL BULLISH | Score 82 | Confidence A-
Reason: positive catalyst + breakout + call activity spike
Contract: next-week ATM call
Entry: only if AAPL stays above breakout level
Invalidation: lose breakout/VWAP
Target 1: trim at +25% option premium
Target 2: exit remainder at +50% or end-of-day
Time stop: close by 3:30 PM ET if no follow-through
Liquidity: spread acceptable
Data note: Tradier live options + Tier 1 news source

==================================================
17. PAPER EXECUTION RULES
==================================================
If paper execution is enabled:
- use limit orders only,
- trade one contract only,
- do not average down,
- do not scale in,
- one open signal per symbol at a time,
- cancel entry if price condition breaks,
- auto-close by time stop if thesis stalls.

Alpaca paper trading remains the default execution simulator unless a later revision explicitly swaps this out.

==================================================
18. REVIEW AND REPLAY
==================================================
This project must support replay.

Replay should answer:
- what information the engine had at a given timestamp,
- which provider supplied each event,
- whether the news label was correct,
- whether price confirmed in time,
- whether options activity truly spiked or was only suggestive,
- whether the contract choice was reasonable,
- whether the order would likely have filled,
- whether the exit logic helped or hurt.

Do not rely only on candle backtests. This strategy depends on event timing, news sequencing, source quality, and options flow.

20. IMPLEMENTATION BEST PRACTICES FOR THE BUDGET STACK
==================================================
A. Prefer streaming where available
- stream market data from the primary provider instead of polling when possible,
- use polling mainly for RSS/feed sources and periodic chain refreshes.

B. Keep ingestion idempotent
- every normalized event should have a deterministic dedupe key,
- retries must not create duplicate alerts or duplicate raw rows.

C. Separate source confidence from signal quality
- a good setup on weak data is not the same as a good setup on strong data,
- keep both scores explicit.

D. Cache aggressively
- cache symbol metadata, expiration calendars, and frequently used chain lookups,
- only refresh heavy endpoints when price/news activity justifies it.

E. LLM budget discipline
- only send shortlisted events to an LLM,
- save prompts/responses for replay,
- prefer small local models first.

F. Make provider failures visible
- track lag, disconnects, empty responses, parse failures, and stale-source windows,
- expose provider health in the review API.

21. WHAT NOT TO USE FOR V1
==================================================
Avoid these as primary V1 choices:
- expensive institutional data stacks before validation,
- X/Twitter sentiment as a primary trigger,
- browser-automation-heavy scraping workflows,
- multi-agent swarms,
- live auto-execution,
- UI-first development.

Reason:
They add complexity or cost without improving the core signal engine enough for V1.
